# Mock分组复制/移动设计说明

## 背景

当前批量选择 Mock 分组后，页面只提供“复制到项目”能力。
如果要支持“移动到项目”，除了交互调整外，还要保证迁移方式不会破坏分组地址、历史链路和下属数据关联。

统一权限基线已独立沉淀到：

[`docs/mock-group-permission-design.md`](/D:/MyCodes/OtherCodes/simple-boot-mock-server07/docs/mock-group-permission-design.md)

复制、移动相关实现必须先满足该文档中的统一权限模型。

## 目标

1. 将现有“复制到项目”升级为统一的“转移到项目”能力
2. 支持两种动作：`copy`（复制）和 `move`（移动）
3. 在不破坏现有分组地址和数据关联的前提下完成迁移

## 复制和移动的权限矩阵

### 复制分组

复制的本质是“读取源分组 + 向目标项目写入新分组”。

权限要求：

1. 源分组需要有 `readable`
2. 目标项目需要有 `writable`

### 移动分组

移动的本质不是“复制后删除”，而是“修改分组归属关系”。
为了符合语义和权限边界，移动应视为“从源项目移走资源并写入目标项目”。

权限要求：

1. 源分组需要有 `deletable`
2. 目标项目需要有 `writable`

## 为什么移动不建议做成“复制后删除”

推荐移动实现为“直接修改分组所属项目”，不要实现成“复制新分组 + 删除旧分组”。

原因：

1. 分组下的请求、数据、场景、Schema 都是通过 `groupId` 关联的，直接修改分组归属即可完成整体迁移
2. 可以保留原始 `groupId`、`groupPath`、历史链路和已有引用关系
3. Mock 地址通常依赖 `groupPath`，移动后保留 `groupPath` 可以避免 URL 变化
4. 复制后删除会生成全新的分组和子数据，审计链路更复杂，也更容易引入遗漏

## 建议的交互改造

建议将当前“复制到项目”弹窗升级为统一弹窗：

1. 标题：`转移到项目`
2. 动作类型：
   - `复制`
   - `移动`
3. 目标用户
4. 目标项目
5. 操作说明

其中：

1. 普通用户移动到默认项目时，目标用户固定为自己
2. 管理员可以切换目标用户，再选择该用户的默认项目或其它项目
3. 当目标项目和源项目完全一致时，应直接提示“目标项目不能与源项目相同”
4. 对批量数据，如果选中的分组来源权限不一致，应在弹窗打开前先做能力判断并禁用不支持的动作

## 建议的前端能力划分

### 列表页按钮

建议拆分为两个明确操作：

1. `批量复制`
2. `批量移动`

单行操作区也建议提供：

1. `复制`
2. `移动`

### 按钮显隐建议

1. `复制` 按钮：
   - 当前用户对源分组有读权限时显示
2. `移动` 按钮：
   - 当前用户对源分组有删权限时显示

### 弹窗提示建议

当动作类型为 `move` 时，增加说明：

1. 将保留原有 `groupPath`
2. 分组下请求、数据、场景会一并迁移
3. 需要对源分组有删除权限，对目标项目有写权限

## 建议的后端接口

建议新增统一接口：

`POST /admin/groups/transfer`

请求体示例：

```json
{
  "groupIds": [1, 2, 3],
  "action": "move",
  "projectId": 12,
  "projectCode": "demo_project",
  "userName": "test"
}
```

字段说明：

1. `groupIds`：待处理分组 ID 列表
2. `action`：`copy` 或 `move`
3. `projectId/projectCode/userName`：目标项目定位信息

## 建议的后端校验顺序

### 第一步：解析目标项目

1. 如果是默认项目，按 `projectCode=default` 处理
2. 默认项目必须明确目标 `userName`
3. 非默认项目以 `projectId` 为主，`projectCode` 为兼容补充

### 第二步：逐个校验源分组

1. 分组是否存在
2. 是否有源权限
3. 是否允许目标迁移
4. 是否出现“移动到同一项目”

### 第三步：校验目标权限

1. 管理员永远通过
2. 默认项目只要目标用户是当前用户本人，则通过
3. 非默认项目需校验 `writable`

### 第四步：统一执行

建议采用“先全量校验，再批量执行”的方式，避免部分成功部分失败导致用户理解成本过高。

## 建议的服务层职责

建议新增统一服务入口，例如：

1. `transferGroups(List<Integer> groupIds, TransferAction action, MockProject targetProject, String targetUserName)`

内部再拆分：

1. `copyGroups(...)`
2. `moveGroups(...)`

### copyGroups

1. 复用现有复制逻辑
2. 补上源分组读权限校验
3. 批量返回处理结果

### moveGroups

1. 基于事务批量更新分组的 `userName`、`projectId`、`projectCode`
2. 默认保留原 `groupPath`
3. 不重建请求、数据、场景、Schema
4. 使用现有审计能力记录变更

## 推荐的数据行为

### 复制

1. 新分组生成新的 `id`
2. 新分组生成新的 `groupPath`
3. 同项目复制时，名称可追加 `-copy`
4. 跨项目复制可保持原名称，若冲突再提示或自动处理

### 移动

1. 保留原 `id`
2. 保留原 `groupPath`
3. 更新 `userName/projectId/projectCode`
4. 保留原请求、数据、场景、Schema 关联
