# Mock分组统一权限设计说明

## 背景

Mock 分组权限目前散落在多个前后端入口里，容易出现不同页面、不同接口各自计算一套权限的问题。
尤其是默认项目和管理员场景，如果判断口径不统一，很容易出现权限遗漏或放大。

本文档用于定义 Mock 分组及其下属资源的统一权限模型。

## 适用范围

本文档适用于所有与 Mock 分组权限相关的场景，包括但不限于：

1. 分组列表是否可见
2. 分组详情是否可访问
3. 分组是否可新建、编辑、删除
4. 分组是否可复制、移动
5. 分组下的请求、数据、场景、Schema 是否可读写删
6. 依赖分组权限进行的其它批量操作

## 统一原则

Mock 分组权限不单独维护独立模型，而是统一从以下条件推导：

1. 是否管理员
2. 是否默认项目且分组属于当前用户本人
3. 非默认项目下的 owner/协作成员权限

判断顺序固定为：

1. 先判断是否管理员
2. 再判断是否默认项目且属于当前用户本人
3. 最后按非默认项目权限判断

## 核心规则

### 最高优先级规则

1. 管理员拥有所有项目、所有分组的全部权限
2. 只要分组属于默认项目，且 `group.userName` 等于当前登录用户，则当前用户对该分组拥有全部权限
3. 默认项目不依赖 `projectUsers` 协作成员表做权限判定

### 默认项目规则

默认项目使用 `projectCode=default` 作为兼容标识，通常没有有效的 `projectId`。

默认项目下的分组权限判定规则必须固定为：

1. `admin` 用户永远有读、写、删、复制、移动权限
2. 普通用户只要该默认项目分组的 `userName` 是自己，就有读、写、删、复制、移动权限
3. 普通用户如果该默认项目分组的 `userName` 不是自己，则无权限

可以理解为：

`默认项目权限 = 是否管理员 || 是否当前用户自己的默认项目分组`

### 非默认项目规则

非默认项目按项目归属和项目协作权限判断：

1. 管理员：全部权限
2. 项目 owner：全部权限
3. 协作成员：
   - `readable` 可读
   - `writable` 可新增和修改
   - `deletable` 可删除

## 统一分组权限映射

分组相关能力统一映射为：

1. 分组可见：需要 `readable`
2. 分组详情可见：需要 `readable`
3. 分组新建：目标项目需要 `writable`
4. 分组编辑：源分组需要 `writable`
5. 分组删除：源分组需要 `deletable`
6. 分组复制：源分组需要 `readable`，目标项目需要 `writable`
7. 分组移动：源分组需要 `deletable`，目标项目需要 `writable`
8. 分组下请求/数据/场景/Schema 查看：跟随分组 `readable`
9. 分组下请求/数据/场景/Schema 编辑：跟随分组 `writable`
10. 分组下请求/数据/场景/Schema 删除：跟随分组 `deletable`

这意味着：

1. 同一分组上的按钮显隐、前端禁用、后端接口校验都应保持一致
2. 不应出现“默认项目分组可编辑，但删除、移动、复制又被额外限制”的情况
3. 不应出现“分组可见，但其下请求/数据/场景又按另一套权限判断”的情况

## 必测场景

### 默认项目

1. 自己默认项目分组可读、可写、可删、可复制、可移动
2. 他人默认项目分组对普通用户不可读、不可写、不可删、不可复制、不可移动
3. 管理员可操作任意默认项目分组

### 非默认项目

1. 只有 `readable` 的协作成员只能查看和复制
2. 只有 `writable` 的协作成员可以编辑，但不能删除和移动
3. 同时有 `deletable + writable` 时可移动
4. owner 可执行所有操作

## 实现注意事项

1. 默认项目权限判断不要依赖 `projectUsers`
2. 默认项目权限判断不要只看 `projectCode=default`，还要同时校验 `group.userName`
3. 管理员权限应在最前面短路返回，避免后续分支误判
4. 任何涉及分组及其子资源权限的地方，都要优先套用本文档规则

## 结论

后续实现 Mock 分组及其相关功能时，权限基线固定为：

1. 管理员拥有全部权限
2. 默认项目下，只要分组属于当前用户本人，就拥有全部权限
3. 非默认项目按 owner/协作成员权限判定
4. 复制使用 `源 readable + 目标 writable`
5. 移动使用 `源 deletable + 目标 writable`
